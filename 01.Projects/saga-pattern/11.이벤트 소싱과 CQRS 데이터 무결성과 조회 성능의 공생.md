이벤트 소싱(Event Sourcing) 아키텍처에서 데이터의 상태를 덮어쓰지 않고 발생한 모든 사건(Event)을 순차적으로 저장하는 방식은 완벽한 감사 로그와 데이터 이력 추적을 가능하게 합니다. 하지만 이 방식은 시스템의 **'조회(Query) 효율성' 측면에서 치명적인 구조적 한계**를 가지며, 이를 극복하기 위해 **CQRS(명령 쿼리 책임 분리) 패턴과의 결합이 필수적**으로 요구됩니다.

이벤트 소싱이라는 더 큰 맥락에서 CQRS와의 결합이 왜 필수적이며 어떻게 작동하는지 구체적으로 살펴보면 다음과 같습니다.

## 1. 이벤트 소싱의 근본적인 한계: 조회(Query)의 비효율성

이벤트 소싱 시스템에서 특정 데이터의 '현재 상태'를 조회하려면, 과거부터 발생한 모든 관련 이벤트를 처음부터 끝까지 다시 재생(Replay)하고 집계(Aggregate)해야 합니다.

- **성능 저하 및 예측 불가:** 저장된 이벤트가 많아질수록 현재 상태를 재구성하는 과정은 매우 느려지고 응답 시간을 예측하기 어려워집니다.
- **복잡한 쿼리의 불가능성:** 일반적인 관계형 데이터베이스처럼 여러 엔티티를 조인(Join)하거나 복잡한 조건으로 검색하는 작업은 순차적인 이벤트 로그(Event Store) 상에서는 합리적인 시간 내에 수행하는 것이 거의 불가능에 가깝습니다.

## 2. CQRS를 통한 해결: 읽기와 쓰기의 완벽한 분리

조회 비효율성 문제를 해결하기 위해 시스템의 데이터를 변경하는 **'쓰기(Command)'**와 데이터를 가져오는 **'읽기(Query)'**의 책임을 물리적/논리적으로 분리하는 CQRS 패턴이 필수적으로 도입됩니다.

- **쓰기 모델 (Command Model = Event Store):** 이벤트 소싱 자체가 CQRS의 쓰기 모델 역할을 담당합니다. 비즈니스 로직을 처리하고 상태 변경을 불변의 이벤트 스트림으로 추가 전용(Append-only) 저장소에 기록하는 데에만 집중합니다.
- **읽기 모델 (Query Model = Projection):** 읽기 모델은 이벤트 스토어에 발행되는 이벤트 스트림을 비동기적으로 구독(Subscribe)합니다. 수신한 이벤트를 바탕으로 사용자 화면이나 쿼리에 최적화된 형태의 **프로젝션(Projection)** 또는 **구체화된 뷰(Materialized View)**를 생성하고 지속적으로 업데이트합니다.

## 3. 결합을 통해 얻는 전략적 이점

이 두 패턴을 결합함으로써 시스템은 이벤트 소싱의 장점을 잃지 않으면서도 강력한 성능 이점을 확보합니다.

- **조회 성능 극대화:** 읽기 모델은 복잡한 연산이나 이벤트 재생 없이, 이미 조회에 맞게 계산 및 가공되어 저장된 구체화된 뷰를 단순히 읽어오기만 하면 됩니다. 이는 빠르고 효율적인 쿼리에 최적화된 **'지속형 읽기 전용 캐시'**처럼 작동합니다.
- **독립적인 확장성 (Scalability):** 대부분의 시스템은 쓰기보다 읽기 요청이 압도적으로 많습니다. CQRS를 통해 분리된 읽기 모델의 인프라만을 독립적으로 스케일아웃(Scale-out)하여 트래픽 불균형에 효율적으로 대응할 수 있습니다.
- **유연한 뷰(View) 재생성:** 이벤트 스토어가 '단일 진실 원본(Single Source of Truth)' 역할을 하므로, 새로운 조회 요구사항(예: 새로운 통계 화면 추가)이 생기더라도 과거의 이벤트를 처음부터 다시 재생하여 새로운 읽기 모델(새로운 프로젝션)을 완벽하게 재구축할 수 있습니다.

## 요약

결론적으로 이벤트 소싱 생태계에서 CQRS는 선택적인 아키텍처가 아니라 **이벤트 소싱의 데이터 조회 및 추출의 비효율성을 해결하기 위한 필수 불가결한 전제 조건**입니다. 이벤트 소싱이 쓰기의 무결성과 데이터의 히스토리를 책임진다면, CQRS는 이를 실시간 서비스에 걸맞은 속도로 가공하고 제공하는 역할을 수행합니다.
